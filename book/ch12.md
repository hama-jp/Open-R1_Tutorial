## 12.1 Countdown ゲームとは

英国のTV番組で有名なパズルで、ルールは単純です。

> **「与えられた 4〜6 個の数字と四則演算だけを使って、目標の数字を作れ」**

例: `[3, 7, 8, 10]` を使って `24` を作れ。 
→ `(10 - 8 + 3) * 7 - ???` のような試行錯誤が求められる。

この問題設定は

- ルールが簡単で **報酬関数がルールベースで書ける**
- 答えが一意とは限らず、**推論が必要**
- 入力が短い（長い CoT にもできる）

という特徴から、**GRPO の教材に最適**です。

## 12.2 データセット準備

```python
# examples/ch12/make_countdown.py
import random, json
random.seed(42)

def gen_problem():
    nums = random.sample(range(1, 100), 4)
    # 適当な式で target を作る（必ず解があるように）
    a, b, c, d = nums
    target = (a + b) * (c - d) if c > d else (a * b) - (c + d)
    return {"numbers": nums, "target": abs(target)}

rows = [gen_problem() for _ in range(2000)]
open("countdown_train.jsonl", "w").write(
    "\n".join(json.dumps(r) for r in rows)
)
```

## 12.3 報酬関数を設計する

2 種類組み合わせます。

### 12.3.1 書式報酬 — `<think>` と `<answer>` タグが正しく現れるか

```python
import re
PATTERN = re.compile(
    r"<think>.*?</think>\s*<answer>(.+?)</answer>", re.DOTALL
)
def format_reward(completion):
    return 1.0 if PATTERN.search(completion) else 0.0
```

### 12.3.2 正解報酬 — 式が **与えられた数字だけ** で組まれ、正しい target を生むか

```python
ALLOWED = set("0123456789+-*/() ")

def eval_reward(completion, numbers, target):
    m = PATTERN.search(completion)
    if not m:
        return 0.0
    expr = m.group(1).strip()
    # 許可文字以外はアウト
    if any(ch not in ALLOWED for ch in expr):
        return 0.0
    # 数字集合が一致するか
    used = list(map(int, re.findall(r"\d+", expr)))
    if sorted(used) != sorted(numbers):
        return 0.0
    try:
        val = eval(expr, {"__builtins__": {}})
    except Exception:
        return 0.0
    return 1.0 if val == target else 0.0
```

> ⚠️ **Warning**  `eval` は本来危険な関数です。
> ここではホワイトリスト文字種に絞ることで **任意コード実行のリスクをかなり下げています** が、
> **「安全である」と言い切るべきではありません**（環境変数・隠れた属性アクセス経路などは常に想定し切れない）。
> 本章はあくまで **教育用の簡略化として `eval` を許容** している位置付けです。
> 本番では `ast.literal_eval` + 自前パーサ、あるいは E2B / Morph のようなサンドボックスに出してください。

### 本章の verifier の「教育用簡略化」の範囲

本章の `eval_reward` / `format_reward` は **教育用の最小例** です。
研究・本番運用の報酬検証器（verifier）としては、最低でも以下のガードを追加する必要があります。

| 観点 | 本章での扱い | 本格運用で必要な対応 |
|---|---|---|
| 式の等価性 | 値の完全一致のみ | `sympy` 等で正規化・可換律考慮 |
| 冗長・未使用トークン | チェックなし | 「式に現れない数字」「使いすぎた数字」を減点 |
| 途中 invalid 生成 | パース失敗で 0.0 | 部分点や再試行導線を検討 |
| タイムアウト | なし | 重い `eval` を沙箱で制限時間付き実行 |
| サンドボックス隔離 | ホワイトリストのみ | プロセス分離 / E2B / Morph sandbox |
| 書式報酬の重み | 固定（0.1） | タスク難度と連動させて動的調整 |

**本章の目的は verifier の完成度ではなく、GRPO の観測単位（同一プロンプトでの G サンプル・報酬・アドバンテージ）を手で掴むこと** です。
このチュートリアルの報酬関数をそのまま研究用途に転用する前に、上記の穴を埋めてください。

## 12.4 GRPO で学習する

```python
# examples/ch12/grpo_countdown.py
from trl import GRPOTrainer, GRPOConfig
from transformers import AutoTokenizer
from datasets import load_dataset

MODEL = "Qwen/Qwen2.5-1.5B-Instruct"
tok = AutoTokenizer.from_pretrained(MODEL)

def build_prompt(ex):
    user = (
        f"以下の数字 {ex['numbers']} を **すべて 1 回ずつ** 使い、"
        f"四則演算で {ex['target']} を作る式を求めよ。"
        "\n思考過程は <think></think> で、最終式は <answer></answer> で囲むこと。"
    )
    return {"prompt": tok.apply_chat_template(
        [{"role": "user", "content": user}],
        tokenize=False, add_generation_prompt=True)}

ds = load_dataset("json", data_files="countdown_train.jsonl", split="train")
ds = ds.map(build_prompt)

def reward_fn(prompts, completions, numbers=None, target=None, **kw):
    scores = []
    for c, n, t in zip(completions, numbers, target):
        s = 0.1 * format_reward(c) + eval_reward(c, n, t)
        scores.append(s)
    return scores

args = GRPOConfig(
    output_dir="out-countdown",
    num_generations=8,
    max_prompt_length=256,
    max_completion_length=768,
    per_device_train_batch_size=2,
    gradient_accumulation_steps=4,
    learning_rate=5e-6,
    beta=0.04,
    bf16=True,
    logging_steps=5,
    max_steps=500,
    use_vllm=True,
)

trainer = GRPOTrainer(
    model=MODEL,
    args=args,
    reward_funcs=[reward_fn],
    train_dataset=ds,
)
trainer.train()
```

### 12.4.1 ハードウェア別の所要時間とメモリ目安

設定は本書のスクリプトをそのまま使った場合の **目安値** です（実測はモデル・vLLM 同居有無・データ難度で大きくぶれます）。

| GPU | モデル | `G` | `max_completion_length` | VRAM 目安 | 500 step 所要時間 | 備考 |
|---|---|---:|---:|---:|---:|---|
| Colab T4 (16GB) | Qwen2.5-0.5B-Instruct | 4 | 384 | 12〜14GB | 4〜6 時間 | `bf16=False, fp16=True`, `use_vllm=False` |
| RTX 3090 / 4090 (24GB) | Qwen2.5-1.5B-Instruct | 8 | 512 | 18〜22GB | 2〜3 時間 | vLLM 同居だと OOM 寸前 — `use_vllm=False` 推奨 |
| A100 40GB | Qwen2.5-1.5B-Instruct | 8 | 768 | 30〜36GB | 1.5〜2 時間 | 本章のデフォルト設定 |
| A100 80GB / H100 | Qwen2.5-7B-Instruct | 16 | 1024 | 60〜70GB | 3〜5 時間 | 本格的な R1-Zero ミニ再現 |
| 8 × H100 80GB | Qwen2.5-7B-Instruct | 16 | 2048 | DeepSpeed ZeRO-3 | 1〜2 時間 | 第11.8 節の構成を流用 |

> ⚠️ **OOM が出たらまず削るべき順番**: ①`G`（半分にすれば VRAM はほぼ半分）→ ②`max_completion_length` → ③ モデルを 1 段小さく → ④ vLLM を切る（同居の場合）。
> `gradient_accumulation_steps` を増やすのは **計算時間と引き換え** にメモリが減る、という関係です。

GPU 1 枚の A100 40GB で、500 step 前後で以下が観察できるはずです。

- 生成長（`completion_length`）が **徐々に伸びる**
- 正解率（`reward/eval_reward`）が 0.1 付近から 0.5〜0.7 に上昇
- 出力中に **"Wait, let me reconsider..."** のような自己修正フレーズが登場

## 12.5 "aha moment" を観察する

学習途中の生成ログをダンプし、次のような応答が観察できれば、本章の目的は達成です。

```
<think>
3 + 7 = 10、10 × 8 = 80。でも目標は 85 だから合わない。
待って、10 の代わりに 10 - 8 = 2、2 × 3 = 6、6 + 7 = 13。やっぱり違う。
もう一度考えよう。(10 - 8) = 2、2 × (7 + 3) = 20、まだ違う…
あ、(10 - 3) × (8 - 7) = 7。惜しい。
(10 * 3) - (8 - 7) = 29。違う。
(8 - 3) × (10 + 7) = 85。これだ！
</think>
<answer>(8 - 3) * (10 + 7)</answer>
```

このような **試行錯誤と自己修正** こそが論文で "aha moment" と呼ばれる現象です。

> ⚠️ **観測と解釈を混同しない**
> 「自己修正っぽい文が出ること」と「本当に探索が改善していること」は同値ではありません。
> 見かけ上の自己修正フレーズが増えても、正答率が上がっていなければ
> **書式だけが強化された疑似メタ認知** の可能性があります。
> 判断するには **`reward/mean` が同時に改善しているか** を必ずセットで確認してください。

## 12.6 学習ログの読み方

`trl` は以下の指標を TensorBoard / wandb に出します。

| 指標 | 見方 |
|---|---|
| `train/loss` | 必ず下がるものでもない（負のアドバンテージ次第） |
| `train/reward/mean` | **最重要**。単調に上がってほしい |
| `train/kl` | 参照からどれくらい離れたか。過大だと方策が壊れている |
| `train/completion_length` | 伸びていれば CoT が育っている |
| `train/reward_std` | 0 に落ちるとグループ内差が無くなり学習停滞 |

## 12.7 よくあるハマりどころ

| 症状 | 原因 | 対処 |
|---|---|---|
| 全員正解 or 全員不正解で学習進まず | 問題が簡単すぎ/難しすぎ | データ難易度を調整 |
| reward 0 から動かない | 報酬が binary で粗い | 書式報酬を薄く足す、部分点導入 |
| KL 発散 | lr が高すぎ、clip 広すぎ | `lr` 下げ、`epsilon=0.1` に |
| OOM | `G` が大きすぎ or `max_completion_length` 過大 | どちらか削る |
| vLLM が反応しない | 別 GPU リソース確保失敗 | `use_vllm=False` に fallback |
