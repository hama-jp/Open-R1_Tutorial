## I. 基礎概念・R1 系モデル

- **推論モデル (reasoning model)** — 回答する前に長い CoT を生成し、途中の思考過程を明示的に通すタイプの LLM。通常のチャットモデルとは「推論トークンに計算資源を割く」点で区別される。（→ [第1章](ch01.md)）
- **思考過程 / CoT (chain-of-thought)** — 問題を解くまでに通る途中推論のテキスト列。本書では reasoning trace／推論軌跡と同義で扱う。出力形式は `<think>...</think>` タグで区切るのが定番。（→ [第1章](ch01.md), [第5章](ch05.md)）
- **R1-Zero** — SFT を一切行わず、純粋な RL（GRPO + ルールベース報酬）だけで推論能力を獲得した DeepSeek のモデル。可読性・多言語混在の問題を持つが、"aha moment" が観察された最初の事例。（→ [第1章](ch01.md), [第8章](ch08.md)）
- **R1** — Cold-start SFT → Reasoning RL → Rejection Sampling + SFT → 最終 RL の 4 ステージで作られた実用型の推論モデル。可読性と指示追従を両立する。（→ [第1章](ch01.md), [第8章 8.5 節](ch08.md)）
- **R1-Distill** — R1 が生成した CoT 付き応答を教師テキストとして、小型モデル（Qwen2.5 や Llama-3）を SFT したシリーズ。7B でも数学・コード系で強い。（→ [第1章](ch01.md), [第9章](ch09.md)）
- **Open-R1** — DeepSeek-R1 の学習レシピ・データを完全オープンに再構築する Hugging Face 主導のプロジェクト。モデル公開ではなく「再現可能性の欠損部を埋める」ことが目的。（→ [第1章](ch01.md)）
- **"aha moment"** — 学習中に "Wait, let me reconsider..." のような自己修正フレーズが自発的に現れる現象。報酬ハック（冗長化）との切り分けが必要で、正答率と `completion_length` を同時に観察して判定する。（→ [第8章 8.4 節](ch08.md), [第12章](ch12.md)）
- **訓練レシピ** — データ・ハイパーパラメータ・学習段階の組み合わせ。R1 系では重み単体ではなくこのレシピこそが価値の中心、というのが Open-R1 の立ち位置。（→ [第1章 1.3 節](ch01.md)）

## II. Transformer の骨格

- **デコーダ型 Transformer** — 左から右に次トークンを生成する causal な Transformer。Attention + FFN のブロックを何十層も積む構造。GPT 系・LLaMA・DeepSeek-V3 など LLM の主流形式。（→ [第2章](ch02.md)）
- **残差接続 (residual connection / skip connection)** — サブレイヤの出力を入力に足し込む `x = x + f(x)` の形。恒等写像から出発でき、深いネットでも勾配が届くため、Transformer を「ブロックを積み重ねる」ことで拡張可能にした根幹。（→ [第2章 2.1 節](ch02.md)）
- **残差ストリーム (residual stream)** — 入力埋め込みから出力まで貫く一本の情報の流れ。各層はそこへ「差分」を書き加える装置として見ると LLM 内部の解釈がしやすい。（→ [第2章 2.1 節](ch02.md)）
- **Pre-LN / Post-LN** — 正規化をサブレイヤの前に置くか後に置くか。Pre-LN は学習が安定しやすく、現代 LLM では標準。（→ [第2章 2.1 節](ch02.md)）
- **サブワード (subword)** — 単語でも文字でもない粒度のトークン化単位。BPE・SentencePiece・Unigram LM などが代表的な手法で、未知語を分割して表せる。（→ [第2章 2.2 節](ch02.md)）
- **埋め込み層 (embedding)** — 整数トークン ID をベクトルに変換する `(vocab_size, d_model)` の行列。ID の行を取り出すだけの lookup 操作。（→ [第2章 2.2 節](ch02.md)）
- **重み共有 (weight tying)** — 入力側の埋め込み行列と出力側の `lm_head`（語彙へ射影）に同じ重みを転置で使い回す工夫。パラメータ節約に加え、表現の一貫性にも寄与する。（→ [第2章 2.2 節](ch02.md)）
- **Self-Attention** — 各位置の表現を「他の位置の表現を重み付き平均したもの」として作る仕組み。離れた位置のトークン同士を 1 ステップで関連付けられる。（→ [第2章 2.3 節](ch02.md)）
- **Q / K / V (Query / Key / Value)** — 入力に別々の線形射影を掛けて作る 3 つのベクトル。Q と K の内積で「どの位置がどれくらい関係ありそうか」のスコアを作り、それで V を加重平均する。（→ [第2章 2.3.1 節](ch02.md)）
- **causal mask** — デコーダ型で未来位置のスコアを $-\infty$ に置くマスク。softmax 後にゼロとなり、各位置が過去しか参照できなくなる。これにより T 個の予測を 1 回の forward で並列に学習できる。（→ [第2章 2.3.2 節](ch02.md)）
- **Multi-Head Attention (MHA)** — 小さな Attention を $h$ 個並列に走らせ、それぞれ別の関係性を拾わせる機構。ヘッドごとに異なる観点の「スコア表」を同時に持てる。（→ [第2章 2.3.3 節](ch02.md)）
- **KV キャッシュ** — 推論時に過去の K, V を再計算せず保存しておく高速化技法。長文生成時のメモリボトルネックでもあり、GQA / MLA など各種の圧縮手法が生まれる理由。（→ [第2章 2.3.3 節](ch02.md)）
- **Grouped-Query Attention (GQA)** — 複数の Q ヘッドで K, V を共有し、KV キャッシュサイズを削減する手法。LLaMA-2 以降で広く採用。（→ [第2章 2.3.3 節](ch02.md)）
- **FFN (フィードフォワード)** — Attention で集めた情報をトークン位置ごとに独立に変換する 2 段の MLP。Attention が「位置間で混ぜる」のに対し、FFN は「位置内で変換する」分業関係。現代 LLM ではパラメータの過半がここに集中する。（→ [第2章 2.4 節](ch02.md)）
- **SwiGLU** — FFN の活性化関数として、GELU の代わりに `SiLU(xW) ⊙ (xV)` というゲート機構を使う方式。LLaMA 以降の標準。（→ [第2章 2.4 節](ch02.md)）
- **LayerNorm** — 各ベクトルから平均を引き標準偏差で割る正規化層。初期 Transformer で使われた。（→ [第2章 2.5 節](ch02.md)）
- **RMSNorm** — 平均を引かず、二乗平均平方根 (RMS) だけで割る簡素化版。gain のみを学習しバイアス無し。LayerNorm と性能差はほぼ無く計算が軽いため、LLaMA 以降で採用される。（→ [第2章 2.5 節](ch02.md)）

## III. MoE（Mixture of Experts）

- **Mixture of Experts (MoE)** — FFN を複数のエキスパートに分け、トークンごとにその一部だけを使う構造。総パラメータを増やしつつ 1 トークンあたりの計算量を抑える。（→ [第3章](ch03.md)）
- **条件付き計算 (conditional computation)** — 入力に応じてネットワークの一部だけを活性化する発想。MoE はその典型例で、「計算コストと表現力を切り離す」という設計原理を指す。（→ [第3章 3.1 節](ch03.md)）
- **ルーター (router)** — 各トークンをどのエキスパートに送るかを決める小さな線形層 + softmax。$g(x) = \mathrm{softmax}(x W_g)$ の形。（→ [第3章 3.2 節](ch03.md)）
- **top-k ルーティング** — ルーターの出力から上位 $k$ 個のエキスパートだけを選んで加重和を取る仕組み。$k$ を小さくすると計算が軽いが表現力が落ち、大きくすると逆になる。（→ [第3章 3.2 節](ch03.md)）
- **Shared Expert（共有エキスパート）** — すべてのトークンが必ず通る共通エキスパート。「全トークンに共通する基礎的特徴量」をここに集約することで、ルーテッドエキスパートが尖った役割に集中できる。DeepSeek-V3 の工夫の一つ。（→ [第3章 3.3.1 節](ch03.md)）
- **Fine-grained Experts（細粒度エキスパート）** — 大きなエキスパートを少数ではなく、小さく切って大量（例: 256 個）用意する設計。組み合わせ数が爆発的に増え、表現の多様性が拡張される（combinatorial flexibility）。（→ [第3章 3.3.2 節](ch03.md)）
- **Auxiliary-loss-free ロードバランシング** — 補助損失でルーター分布を均すのではなく、エキスパートごとのバイアス $b_i$ を使用量に応じて動的調整する方式。タスク損失を汚さずにロードを均せるのが利点。（→ [第3章 3.3.3 節](ch03.md)）
- **ルーティング崩壊 (routing collapse)** — 人気エキスパートにトークンが集中し、大半のエキスパートが使われなくなる現象。正のフィードバックで悪化するため、能動的なロードバランシングが必要。（→ [第3章 3.3.3 節](ch03.md)）
- **Expert Parallelism / All-to-All 通信** — エキスパートを複数 GPU に分散配置するとき、トークンを行き先 GPU へ送り計算後に戻すための集団通信パターン。MoE 学習の分散特有のボトルネック。（→ [第3章 3.4 節](ch03.md)）
- **総パラメータ vs 活性パラメータ** — MoE 固有の区別。例: DeepSeek-V3 は総 671B に対し、1 トークンあたり実際に使われる活性パラメータは 37B（約 5.5%）。メモリは総パラメータ分必要だが計算は活性分で済む。（→ [第3章 3.5 節](ch03.md)）

## IV. 位置エンコーディング

- **位置エンコーディング (Positional Encoding)** — Self-Attention が元来順序不変なため、位置情報を別途注入する仕組み。絶対位置（加法的）と相対位置（RoPE 等）の 2 系統がある。（→ [第4章 4.1 節](ch04.md)）
- **RoPE (Rotary Positional Embeddings)** — Q, K のベクトルを位置 $m$ に応じて複素平面で回転させる相対位置エンコーディング。内積が相対位置 $(m-n)$ の関数になるため、順序情報と相対距離を自然に両立する。現代 LLM の標準。（→ [第4章 4.3 節](ch04.md)）
- **Position Interpolation (PI)** — 学習時より長い系列を扱うため、位置を係数 $s<1$ で縮めて学習範囲内に収める RoPE 拡張手法。（→ [第4章 4.5 節](ch04.md)）
- **NTK-aware** — RoPE の低周波成分はそのまま、高周波成分だけを圧縮して外挿性を改善する手法。PI より性能劣化が少ない。（→ [第4章 4.5 節](ch04.md)）
- **YaRN** — NTK-aware に attention の温度補正を加えた手法。32K→128K のような大幅なコンテキスト拡張で最も一般的。DeepSeek-V3 / R1 の 128K コンテキストを支える。（→ [第4章 4.5 節](ch04.md)）
- **Multi-head Latent Attention (MLA)** — DeepSeek 独自の Attention 変種。Q/K/V を低ランク潜在空間に圧縮してから展開することで、KV キャッシュを劇的に縮小。RoPE と共存させるため K を「潜在部分」と「RoPE 専用部分」に分ける設計が特徴。（→ [第4章 4.6 節](ch04.md)）

## V. 事前学習・SFT

- **Pre-training（事前学習）** — 大規模コーパス（例: 14.8T トークン）に対する次トークン予測でベースモデルを作る段階。LLM のほぼ全ての能力の基盤がここで形成される。（→ [第5章](ch05.md)）
- **Base Model** — 事前学習のみを終え、指示追従や CoT を学んでいないモデル。SFT や RL の出発点。（→ [第5章](ch05.md)）
- **Supervised Fine-Tuning (SFT)** — 指示と望ましい応答のペアで教師あり学習する段階。応答側だけに CE loss を掛ける形が一般的。（→ [第5章](ch05.md)）
- **Chat Template** — ユーザ発話とアシスタント応答を単一の文字列にフォーマットする規約。推論モデルでは `<think>` タグを含む形にする。（→ [第5章 5.5 節](ch05.md)）
- **`<think>` タグ** — 思考過程と最終回答を分離するためのマーカー（例: `<think>...</think>回答`）。蒸留 SFT ではここを保持することが本体であり、削ると普通の SFT に退化する。（→ [第5章](ch05.md), [第8章 8.2 節](ch08.md), [第9章](ch09.md)）
- **gradient_checkpointing** — 順伝播の活性を一部捨てて逆伝播時に再計算することで、メモリを時間と引き換えに節約する技法。長い CoT を扱う SFT / RL で必須。（→ [第5章 5.6 節](ch05.md)）

## VI. 強化学習・PPO

- **方策 (policy, $\pi_\theta$)** — 状態から行動分布を返す関数。LLM ではそのまま「次トークン分布 $p_\theta(y_t \mid y_{<t}, x)$」 に一致する。（→ [第6章 6.1 節](ch06.md)）
- **参照方策 ($\pi_\text{ref}$)** — 学習中の方策が離れすぎないよう基準にする固定モデル。通常は SFT 直後のモデル。KL ペナルティの片側に入る。（→ [第6章 6.5 節](ch06.md)）
- **状態・行動・エピソード** — RL 用語を LLM に当てはめた対応。状態 $s$ はプロンプト＋ここまでの生成列、行動 $a$ は次トークン、エピソードは 1 応答の完成。（→ [第6章 6.1 節](ch06.md)）
- **報酬 (reward, $r$)** — 応答生成の完了時に与えるスカラー。良し悪しの信号源で、RM・ルール・実行結果など何でも良い。（→ [第6章 6.1 節](ch06.md), [第8章](ch08.md)）
- **Policy Gradient Theorem** — 方策勾配の基本式。「報酬が高い行動の確率を上げ、低い行動の確率を下げる」を数式化した $\nabla_\theta J = \mathbb{E}[R\,\nabla_\theta \log \pi_\theta(a\mid s)]$。REINFORCE・PPO・GRPO すべての土台。（→ [第6章 6.3 節](ch06.md)）
- **価値関数 (value function, $V_\psi$)** — 「この状態から先、平均でどれくらいの報酬がもらえるか」の見積もり装置。PPO ではアドバンテージのベースラインに使うが、LLM サイズだと学習が不安定になる。GRPO が削除する対象。（→ [第6章 6.3 節](ch06.md)）
- **アドバンテージ (advantage, $A$)** — 報酬からベースラインを引いた量 $A = R - b$。「予想より良かった分（サプライズ）」を表し、方策勾配の分散を下げる。（→ [第6章 6.3 節](ch06.md), [第7章](ch07.md)）
- **PPO (Proximal Policy Optimization)** — 確率比 $\rho = \pi_\theta / \pi_{\theta_\text{old}}$ をクリップして「1 ステップで方策が大きく動きすぎない」ようにする方策勾配法。LLM の RLHF で長く標準だった。（→ [第6章 6.4 節](ch06.md)）
- **クリッピング（$\varepsilon$）** — $\rho$ を $[1-\varepsilon, 1+\varepsilon]$ に制限し、アドバンテージと掛けた値のうち小さい方（pessimism）を採る操作。典型値は 0.1〜0.2。（→ [第6章 6.4 節](ch06.md)）
- **KL 正則化 / KL ペナルティ（$\beta$）** — 参照方策 $\pi_\text{ref}$ との KL 距離を損失に足し、モデルが不自然な方向に走るのを防ぐ。$\beta$ が小さすぎるとモデルが壊れ、大きすぎると学習が進まない（典型 0.01〜0.04）。（→ [第6章 6.5 節](ch06.md)）
- **RLHF (Reinforcement Learning from Human Feedback)** — SFT → 報酬モデル学習 → PPO の 3 ステップで人間の好みに沿わせる手法。InstructGPT / ChatGPT で有名に。（→ [第6章 6.2 節](ch06.md)）
- **報酬モデル (RM)** — ペア比較データから学習した「応答にスカラーを返すニューラルネット」 $r_\phi(x, y)$。Bradley-Terry モデルに基づく対比的クロスエントロピーで学習する。（→ [第6章 6.2 節](ch06.md)）

## VII. GRPO

- **GRPO (Group Relative Policy Optimization)** — PPO から価値関数を削除し、同一プロンプトから取った $G$ 個のサンプル内の報酬統計でアドバンテージを作る RL アルゴリズム。R1 / R1-Zero の訓練に使われた。本書の中心的話題。（→ [第7章](ch07.md)）
- **グループ内標準化** — $A^{(i)} = (R^{(i)} - \mu) / (\sigma + \epsilon)$ の式でグループ内の平均と標準偏差で報酬を正規化する操作。価値関数の代わりに「同じプロンプト内で相対的に良かったか」を評価する。グループ内に差が無いと学習信号がゼロになる点に注意。（→ [第7章 7.2 節](ch07.md)）
- **グループサイズ $G$** — 1 プロンプトあたりの同時サンプル数。大きいほど分散が下がり安定するが計算コストが増える。典型は 4〜64。課題難易度とモデル能力のマッチによって最適値が変わる。（→ [第7章 7.2 節](ch07.md), [第7章 7.7 節](ch07.md)）

## VIII. 報酬設計・創発

- **ルールベース報酬** — 学習済み RM ではなく、正規表現・実行結果・真理値チェックなどの決定論的ルールで与える報酬。R1-Zero の核心。報酬ハックや人手コストの問題を回避できる。（→ [第8章](ch08.md)）
- **正解報酬 (accuracy reward)** — 「最終解答が正解と一致するか」で 0/1 を返す報酬。数学なら `\boxed{...}` の抽出、コードならテストケース実行、論理なら真理値比較で判定。（→ [第8章 8.2.1 節](ch08.md)）
- **書式報酬 (format reward)** — 出力が `<think>...</think>回答` のような指定フォーマットを満たすかで与える補助的報酬。重みを付けすぎると空虚な CoT に最適化される落とし穴あり。正解報酬を主、書式報酬を補助にするのが鉄則。（→ [第8章 8.2.2 節](ch08.md)）
- **報酬ハック (reward hacking)** — 本来の目的を満たさず、報酬関数の抜け穴だけを突く出力に方策が収束する現象。書式だけ満たす空虚な CoT、冗長化だけが進む"偽 aha moment" などが典型例。（→ [第6章 6.5 節](ch06.md), [第8章](ch08.md)）
- **創発 (emergence)** — 単純な局所ルール（例: 正解一致の 0/1 報酬）から、予想以上に高次の振る舞い（自己修正、別解試行）が生まれる現象。本書では魔法ではなく、観測事実と解釈を分けて扱うべきものとして定義する。（→ [第8章 8.4 節](ch08.md)）
- **Cold-start SFT** — R1 の 4 ステージのうち最初のステージ。数千件の整った CoT データで Base Model を予熱し、RL の探索を始めやすくする役割。R1-Zero の可読性問題を解消する。（→ [第1章](ch01.md), [第8章 8.5 節](ch08.md)）
- **Rejection Sampling + SFT** — R1 の 4 ステージのうち 3 番目。中間モデルに大量サンプルさせ、正答したものだけを残して再度 SFT する手法。蒸留データ作成の基本技でもある。（→ [第1章](ch01.md), [第8章 8.5 節](ch08.md), [第9章 9.2 節](ch09.md)）

## IX. 蒸留

- **知識蒸留 (Knowledge Distillation, KD)** — 2015 年 Hinton らが提案した、大型モデル（教師）の知識を小型モデル（生徒）に移すフレームワーク。古典的には教師・生徒のソフト分布を KL で近づける。（→ [第9章 9.1 節](ch09.md)）
- **sequence-level distillation** — 教師モデルが生成したテキストをそのまま SFT データに使う蒸留形式。ロジット蒸留と比べ通信・保存コストが劇的に軽く、LLM では事実上の標準（単に "Distill" と呼ぶことが多い）。（→ [第9章 9.1 節](ch09.md)）
- **探索方策の転写** — 蒸留は「答えの要約」ではなく、教師が `<think>` 内に残した「どこで分岐し、どこで行き詰まり、どう立て直したか」の探索軌跡を小型モデルに条件付き分布として移植する操作、という本書での捉え方。（→ [第9章 9.1 節](ch09.md)）

## X. 評価

- **pass@1 / pass@k** — $k$ 回サンプルしたうち少なくとも 1 つ正答する確率。pass@1 は「1 回勝負の成功率」（実運用イメージ）、pass@k ($k>1$) は「何度でも引き直せる研究用途・Best-of-N の上限」 に対応する。同じモデルでも $k$ 次第で見た目が大きく違うため、数字を読むときは必ず $k$ を確認する。（→ [第10章 10.2 節](ch10.md)）
- **温度 (temperature) / top_p** — サンプリング時の分布の尖らせ方を制御するハイパーパラメータ。DeepSeek-R1 公式推奨は `temperature=0.6, top_p=0.95`。greedy（0.0）は再現性重視、高温は多様性重視。（→ [第10章 10.4 節](ch10.md)）
- **Contamination（データ汚染）** — 評価ベンチマークの問題文・答えが事前学習データに混ざり込む現象。モデルが「推論している」ように見えて暗唱しているだけ、という事態を引き起こす。高い評価値は「暗記 + 汎化」の合計である点に注意。（→ [第10章 10.5 節](ch10.md)）
- **n-gram 一致** — 評価データと学習データの n-gram 重複率で contamination を事前検知する素朴な手法。パラフレーズや数値バリアントでのクロスチェックと併用するのが実務的。（→ [第10章 10.5 節](ch10.md)）

## XI. 実装・実験

- **trl / vllm / lighteval** — Open-R1 の主要依存ライブラリ。`trl` は SFT・PPO・GRPO トレーナを提供する RL ライブラリ、`vllm` はサンプリングを高速化する推論エンジン（GRPO では学習ループの外でサンプラー専用に起動する）、`lighteval` は pass@k 等のベンチマーク評価を統一的に走らせるツール。（→ [第7章 7.6 節](ch07.md), [第10章](ch10.md), [第11章](ch11.md)）
- **DeepSpeed ZeRO-3** — オプティマイザ状態・勾配・モデルパラメータを複数 GPU に分散保持するメモリ節約技法。24GB クラスの GPU でも数 B パラメータの学習を可能にする。Open-R1 でも多用される。（→ [第11章 11.8 節](ch11.md)）
- **Countdown ゲーム** — 4〜6 個の数字と四則演算だけを使って目標値を作るパズル。与えられた数字だけを使う制約と正解判定器の作りやすさから、GRPO のミニ実験の教材として最適。第12章のハンズオンで使用する。（→ [第12章](ch12.md)）
